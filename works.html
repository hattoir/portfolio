ミーミルの手

バーコード認識・自動仕分けシステム
詳細資料 (PDF) を開く GitHub (後日公開)

【システム概要】

荷物に貼付されたバーコードをカメラで読み取り、その情報に基づいて3軸ロボットアームが指定の場所へ荷物を自動仕分けするシステムです。ベルトコンベア上で荷物を搬送し、読み取り位置でコンベアを正確に停止させてバーコード情報を取得。そのデータをメインマイコンであるESP32へ一度だけ確実に送信し、サーボモーターを駆動させてアームで仕分けを行う一連のFAプロセスを構築しました。

【使用技術・ツール】

  • 機構設計（CAD）： Fusion 360（ロボットアーム本体、ベルトコンベア）
  • 制御マイコン・回路： ESP32、Arduino IDE
  • アクチュエータ： サーボモーター

【ハードウェア設計における課題と工夫】

Fusion 360を用いた実機設計において、画面上のモデリングと現実の物理的な組み上がりのギャップに最も苦心しました。特に、3Dプリンター等で出力した際の「公差の設定」や、各リンク機構が干渉しない適切な「サイズ感の決定」、および複数の部品をガタつきなく強固に結合するための「組み付け構造の設計」において、何度もプロトタイピングと修正を繰り返して完成度を高めました。

【反省点と今後の展望（Next Action）】

今回の開発を通して、以下の技術的な改善点を見出し、次期モデルへのアップデート目標としています。

1. 逆運動学（Inverse Kinematics）の実装による拡張性の確保

現在は各関節の角度を直接指定する制御（順運動学的なアプローチ）となっているため、仕分け先のバーコード種類が増えるたびに、新しい座標への移動コードを手動で追加する必要があります。これを根本から解決するため、目標のXYZ座標を与えれば各関節の角度を自動計算する「逆運動学」を用いたアルゴリズムに改修し、拡張性と保守性の高いシステムを目指します。

2. 逆起電力を考慮した安定した回路設計

回路設計時、サーボモーターの駆動や停止に伴って発生する逆起電力（ノイズ）により、マイコンやモーターが予期せぬ誤動作を起こすというハードウェア特有のトラブルを経験しました。今後の基板設計では、ダイオードやバイパスコンデンサなどの保護部品を適切に配置し、電源系統の分離を行うなど、電気的に安定した堅牢な回路設計を徹底します。

3. エンドエフェクタ（ハンド部）の機構設計とギア精度の向上

アーム先端で荷物を掴むハンド部分の機構設計において、動力を伝達するギアの調整に反省点が残りました。CAD上での設計公差や、実際のギアの噛み合い（バックラッシュ）の調整が甘かったため、モーターのトルクが適切に伝達されず、対象物をうまく把持（グリップ）できないケースが発生しました。この物理的なハードウェア特有の課題を踏まえ、今後の設計ではギアのモジュールや軸間距離をより厳密に計算し、3Dプリンター等の出力精度も加味した上で、確実で安定した把持ができるエンドエフェクタ機構へとアップデートを図ります。

自動運転ミニカー（製作途中）

特定のコースを自走で運転するミニカー
Notion 開発日記・技術レポート GitHub リポジトリ

【システム概要】

本プロジェクトは、ROS 2、Gazebo、およびPyTorchを統合し、カメラ画像から車両の運動制御（cmd_vel）を直接ダイレクトに推論するEnd-to-Endのディープラーニング自動運転ミニカーシステムです。単眼カメラから得たリアルタイム映像を軽量なCNNモデル（DrivingCNN）に入力し、走行レーンを判断して自律走行を行います。シミュレーション環境での検証（Sim）を経て、実機（Real）への移植（Sim-to-Real）を見据えた回路設計・実装までを一貫して行うことを目的としています。

【使用技術、ツール】

  • ロボット制御ミドルウェア： ROS 2 (Jazzy Jalisco)
  • シミュレーション環境： Gazebo Sim (Version 8.x), Antigravity CLI
  • AI / ディープラーニング： PyTorch (with CUDA), OpenCV
  • ハードウェア開発： Arduino IDE, KiCAD（予定）
  • 使用部品： Raspberry Pi 5, ODK-Dカメラ, サーボモーター, DCモーター, モータードライバー, XHコネクタ, ユニバーサル基板

【ハードウェア設計における課題と工夫】

実機試作に向けた初期ハードウェア開発において、以下の課題に直面し、それぞれ対策を講じました。

電源系統の設計ミスと回路修正

課題： ユニバーサル基板の初期設計段階で、モータードライバー用の電源ラインが不足している設計ミスを発見。
工夫： 回路図・実装図の段階でミスを検知し、コネクタを1つ追加することで対応。今後はこの経験をもとに、より可視化・検証がしやすいKiCADを用いた回路設計へと移行します。

確実な配線実装と接続信頼性の向上

課題： 振動の多いミニカーにおいて、配線の接触不良は致命的なトラブルの原因となる。また、スズメッキ線を用いた手作業でのハンダ付けは熱伝導の制御が難しく、イモハンダのリスクがあった。
工夫： すべての配線にXHコネクタを採用し、専用工具で端子を圧着することで物理的な接続信頼性を確保。ハンダ付けにおいては「予備ハンダの徹底」と「スズメッキ線をしっかり熱して熱効率を上げる」ことを意識し、堅牢な基板へと仕上げました。

今後の機体レイアウト最適化

課題： 走行時の安定性やメンテナンス性を考慮する必要がある。
工夫： ラズパイ5やカメラ、シャーシの配置をCAD上でモデル化（または既存モデルの選定）し、重量バランス（重心）を意識したパーツ配置と、整備しやすい配線レイアウトを設計中です。

【反省点と今後の展望（Next Action）】

シミュレーション環境におけるEnd-to-End学習および制御アルゴリズムの構築において、以下の4つの技術的課題を見出し、それぞれ具体的なアプローチで対策・改善を行いました。

1. データ不均衡によるローカルミニマム（その場旋回ループ）の解消

反省点： 初期学習後のテスト走行時、ミニカーが同じ場所を行ったり来たり旋回する挙動（ローカルミニマム）が発生しました。カメラ画像から目先の操作（右・左）だけを予測するEnd-to-End学習の特性上、「自分が今コースのどこにいるか」というマクロなルールを認識できず、さらにデータ収集時に含まれた「その場旋回データ」を過学習したことが原因でした。
Next Action： データクレンジングを自動化し、車両停止状態（$v_x = 0.0$）での旋回データ（$|\omega_z| > 0.5$）や完全停止データを100%除外して再学習を行いました。結果、バイアスが排除され、低速ながらも自力でコースを1周することに成功しました。

2. システム同定（SysID）に基づくPID制御の導入とパラメータ適合

反省点： AIの出力のみ（End-to-End単体）では、速度を上げた際にカーブが不安定になり、完璧な学習データを用意し続けることの限界に直面しました。そこでAIには「目標値（目標速度・目標角度）」だけを出力させ、実際のタイヤ制御はPIDで行うハイブリッド方式を試みましたが、導入初期は車体がその場旋回するか壁に衝突する状態になりました。原因を追究したところ、機体の総質量（6.14kg）をはじめとするURDF（物理モデルパラメータ）に大きな誤差があり、ゲイン計算の前提が崩れていたことが判明しました。
Next Action： 車体重量の誤差を修正し、システム同定から得られた物理特性（無駄時間 $L=0.1$秒、時定数 $T=0.33$秒）をベースに、ジグラー・ニコルス法等を用いて以下の通りゲインを再計算・適合させました。 ・比例ゲイン ($K_p$): 3.92
・積分ゲイン ($K_i$): 0.65 （制御周期 30Hz換算）
・微分ゲイン ($K_d$): 6.0 （制御周期 30Hz換算） これにより、制御が破綻することなく高速化の土台を作ることができました。

3. 急激な車速変化に伴う慣性への対処（ローパスフィルターと減速ロジック）

反省点： PID制御とURDFの加速度制限の上限を3倍に引き上げたことで、直線での最高速度は大幅に向上したものの、1つ目のカーブを抜けた直後にステアリングに大きな数値が残ったまま車速が 3.0 m/s へ急上昇し、慣性によって制御不能（オーバーステア）となり壁に衝突する問題が発生しました。
Next Action： 急激なステアリング変化を滑らかにするため、操作量にローパスフィルターを導入しました。あわせて、カーブ進入時に自動で減速するシステムを組み込みました。さらに、従来のステアリング式「(0.4 × 前回の値) + (0.6 × AIの予測)」の配分を見直し、AIの予測の重みを 0.85 まで引き上げることで、ラインから外れそうになった際のリカバー力を高め、壁に激突せず1周できるよう改善しました。

4. 高速周回時の膨らみ抑制と振動（ハンチング）の防止

反省点： さらなる高速化を目指し、「未来予測の先行入力（車速に応じて早めにハンドルを切る）」を導入したところ、逆にカーブ前の段階でイン側に膨らみすぎて挙動が不安定化し、外側の壁に張り付くような走行ラインになってしまいました。また、実機を見据えた際、サーボモーターの突っ張り（過剰な応答）によるメカの破損リスクが懸念されました。
Next Action： 逆効果となっていた未来予測の先行入力を一度削除し、定常位置を真ん中に維持するため、固有角周波数 ω_n = 5.0、減衰比 ζ = 1.0 に再調整した以下のステップ応答ゲインへ上書きしました。 ・K_p_steer = 25.0 / K_d_steer = 10.0
・直線部最高速度上限: 3.5 m/s
・TCS（トラクションコントロール）作動時制限速度: 1.4 m/s このチューニングにより、サーボの突っ張りを防ぎながら、外壁に擦ることのない美しいラインでの「高速無限周回」の実現へとアップデートを継続しています。
